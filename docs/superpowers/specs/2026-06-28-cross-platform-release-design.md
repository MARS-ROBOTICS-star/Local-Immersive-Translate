# Cross-Platform Release and Local Backend Deployment Design

## Goal

Publish this Zotero plugin from `MARS-ROBOTICS-star/Local-Immersive-Translate`
so users on Windows, macOS, and Linux can install the Zotero `.xpi` from GitHub
Releases and run the local BabelDOC backend without editing hard-coded paths.

## Scope

- Replace public links and package metadata that still point to the original
  upstream repository.
- Remove the in-plugin issue feedback action and its user-facing text.
- Keep Zotero internal identifiers stable unless a future migration requires a
  deliberate plugin identity change.
- Add a networked local backend installer for each platform:
  - macOS/Linux: `install-local-backend.sh`
  - Windows: `install-local-backend.ps1`
- Make plugin defaults portable by auto-detecting project and `uv` paths instead
  of defaulting to a developer-machine path.
- Document a user workflow that installs the `.xpi`, runs the installer, enters
  model API settings, and starts/tests the local backend from Zotero.

## Non-Goals

- Do not bundle the 1.4 GB `BabelDOC` checkout into the Zotero `.xpi`.
- Do not build native installers, system services, or tray applications.
- Do not require users to build the Zotero plugin from source.
- Do not remove manual path overrides from the preferences UI.

## Release Model

GitHub Actions publishes plugin assets when a tag like `v0.0.24` is pushed.
The release should include the `.xpi` and update manifest generated by
`zotero-plugin-scaffold`.

The release workflow will use the repository-provided `GITHUB_TOKEN`, install
dependencies with pnpm, run the plugin build, and run `pnpm release`. CI should
upload `dist` artifacts because the project config writes build output there.

## Local Backend Install Model

The installer is a separate script from the `.xpi`. It downloads or updates the
project into a standard per-user directory:

- Windows: `%USERPROFILE%\Local-Immersive-Translate`
- macOS/Linux: `$HOME/Local-Immersive-Translate`

The installer checks for Git and `uv`. If `uv` is missing, it installs or
bootstraps `uv` using the platform-supported method. It then ensures the
`BabelDOC` directory and Python dependencies are ready enough for
`uv --directory <project>/BabelDOC run python <project>/local_babeldoc_server/server.py`.

`uv` is the Python environment and dependency runner used by this project. It is
not the main source of disk usage; BabelDOC and its Python dependencies are much
larger. Normal users should not need to understand or configure `uv` manually.

The installer prints the detected project directory and `uv` executable path for
diagnostics. Users only paste these values into Zotero if they intentionally
installed the backend somewhere non-standard or if automatic detection fails.

## Plugin Runtime Detection

The plugin should normalize local backend preferences without writing machine-
specific defaults. When existing values are empty or still contain the previous
developer path, it should try candidates in this order:

1. Existing user preference, if it points to a readable project.
2. `$HOME/Local-Immersive-Translate`.
3. The current development checkout path, only in development builds if safely
   detectable.

For `uv`, the plugin should try:

1. Existing user preference, if readable.
2. Executables found on `PATH` via common names: `uv` and `uv.exe`.
3. Common user install locations:
   - Windows: `%USERPROFILE%\.local\bin\uv.exe`
   - macOS/Linux: `$HOME/.local/bin/uv`

If detection fails, the preferences UI keeps the fields editable and the
`Start / Test` action reports which path could not be found.

## Preferences UI

The existing text fields remain for manual override. Labels and help text should
describe portable defaults rather than a developer machine path. The local
backend section should make it clear that users normally run the installer first,
then click `Start / Test`.

The `Project directory` and `uv path` fields are advanced fallback controls.
They are not part of the normal installation path.

## Feedback Removal

The task manager removes the `Issue Feedback` button and its click handler.
Locale strings and README guidance that ask users to copy a task ID for
feedback are removed or rewritten as neutral diagnostics text.

The preferences About area keeps a GitHub/documentation link to the new
repository but no dedicated feedback feature.

## Documentation

README should explain:

1. Download and install the latest `.xpi` from GitHub Releases.
2. Run the platform installer.
3. Open Zotero plugin preferences.
4. Confirm local backend path detection or paste the printed paths.
5. Enter model API settings.
6. Click `Start / Test`.
7. Translate PDFs from the Zotero item context menu.

Developer docs should avoid absolute local paths and use placeholders or
platform-specific examples.

## Validation

- `pnpm build`
- `pnpm lint:check` if formatting changes are broad enough to justify it
- Manual grep confirming no original upstream repository links remain.
- Manual grep confirming no developer-machine default path remains in plugin
  defaults or docs.
